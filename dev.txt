  SECTION 4: NEW IDEAS & FEATURES TO ADD 💡
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  IDEA-01: CVE Pattern Tag Engine
    - When a finding matches known CVE pattern (Dokans, Vite, Mautic, Moodle, Laravel)
      auto-label the finding: "Matches CVE-YYYY-XXXXX pattern"
    - Huge value for client reports — real documented vuln not theoretical

  IDEA-02: Finding "Risk Score" Dashboard Widget
    - Calculate a project risk score (0-100) like CVSS score for entire project
    - Visual ring/gauge on dashboard showing total exposure level
    - Color: green=safe, yellow=moderate, red=critical, DARK RED=evacuate

  IDEA-03: Passive Recon "Ghost Mode"
    - A mode that ONLY uses dorks + passive reads, never sends a probe to target
    - No footprint on target at all — 100% stealth
    - Useful for initial recon before client has signed off fully

  IDEA-04: Target Import from File
    - Upload a .txt or .csv with list of domains/IPs
    - Auto-normalize and add to project targets in bulk
    - Needed for datacenter clients with 100+ assets

  IDEA-05: "Live Threat Intel" Badge on Findings
    - When .env exposure found, show badge: "Actively exploited by bots at scale"
    - Reference GreyNoise / Shodan threat intel context
    - Makes reports way more convincing for clients who need to see urgency

  IDEA-06: Scope Lock System
    - Define scope boundaries per project: only scan *.example.com
    - Crawler/engine auto-rejects any URL outside defined scope
    - Prevents accidental out-of-scope scanning

  IDEA-07: Screenshot Capture on Critical Findings
    - When critical finding found (exposed .env, SQL dump etc.)
    - Headless browser captures screenshot of the URL as evidence
    - Auto-attach to report like a bug bounty writeup

  IDEA-08: Findings Status Kanban Board
    - Open → Confirmed → Mitigated → False Positive
    - Drag and drop cards per finding
    - Track remediation progress per project

  IDEA-09: Per-Scan Diff Mode
    - Compare current scan vs previous scan findings
    - Show: NEW findings (red), FIXED findings (green), UNCHANGED (grey)
    - Critical for recurring client assessments

  IDEA-10: "Dork Assistant" Chatbot Panel
    - In-app panel where you type a target type (e.g. "WordPress hosting")
    - Returns top 10 dork queries to run right now
    - Smart dork suggester based on stack fingerprint

  IDEA-11: Mobile PWA Mode
    - Make the app installable as a PWA on Android/iOS
    - Offline-capable for viewing saved findings without server

  IDEA-12: Header Fingerprint Database
    - Maintain a mini DB of server/framework response header signatures
    - Auto-detect: Laravel, WordPress, Django, Rails, ASP.NET, Spring Boot
    - More accurate stack fingerprinting = better module selection

  IDEA-13: WebSocket Live Job Feed
    - Replace polling-based job status with WebSocket
    - True real-time streaming logs like a terminal with zero lag

  IDEA-14: Quick-Launch Scan Templates
    - Pre-built scan configs: "WordPress Full Audit", "API Backend Check", "Laravel Env Hunt"
    - One click to run a purpose-built scan package

  IDEA-15: OWASP Top 10 2025 Full Coverage Checklist
    - Visual checklist per project showing which OWASP Top 10 categories are tested
    - Green checkmark = module covers it, Red X = not tested yet
    - Great for client deliverables showing coverage

  IDEA-16: Subdomain Enumeration Module
    - Passive: Certificate Transparency logs (crt.sh API)
    - Passive: DNS brute-force wordlist (top 500 subdomains)
    - Auto-add discovered subdomains as new targets in same project

  IDEA-17: API Endpoint Discovery Module
    - Look for Swagger/OpenAPI spec at /api-docs, /swagger.json, /openapi.yaml
    - Auto-import all endpoints as scan targets
    - Run injection/auth modules against each endpoint

  IDEA-18: Custom Payload Library Manager
    - UI to add/edit/remove payloads per module (SQLi, XSS, traversal etc.)
    - Import/export payload lists (compatible with SecLists format)

  IDEA-19: Notification System
    - Email or webhook alert when critical finding is discovered
    - Slack/Discord webhook support (one config line)

  IDEA-20: Evidence Vault
    - Encrypted local storage for all raw HTTP evidence
    - Full request/response stored for legal proof
    - "Lock" a project to freeze evidence at point-in-time for reporting

  TWO PARALLEL CODE TREES EXIST — NEEDS RECONCILIATION:
    - /src/core/  = newer modular engine system (engine.js, modules, registry)
    - /backend/   = older Express server + engine-bridge.js approach

  Recommended unification path:
    - backend/engine-bridge.js should IMPORT from src/core/engine.js
    - src/core/ becomes the single source of truth for all scan logic
    - backend/ stays as the HTTP API wrapper only

  CORS Strategy:
    - Browser-only scanning = limited by CORS sandbox
    - Recommended: local Node agent on port 7777 OR full VPS backend
    - engine.js + httpClient.js adapter design already supports both modes
